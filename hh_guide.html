HH guide (version 0.4 beta)

HH ('HaHa') is a markup language that may be not so useful. It is based on the assumption that the user knows how HTML works (elements, attributes, ...) and does not want to spend much effort in learning other markup languages. That's 0.074 % of the world population. Less is more .
The HH-reader hh.py (url 1) reads HH-source and produces valid HTML5. Batteries included: automatic numbering of headers and footnotes, metadata extraction from content, printer-friendly url-list, math, highlighting, x3d, ...

1 Basic rules

Opening tag: <p> is reduced to <p

The closing tag can be a nameless >

Attributes in <p class="red" id="example"> are reduced to <p,class=red,id=example

Some attribute-names have a short alternative:

  • c= equals class=
  • i= equals id=
  • h= equals href=
  • s= equals src=
  • a= equals alt=
  • t= equals title=
  • l= equals lang=

Our example becomes: <p,c=red,i=example

If the value of an attribute contains multiple words, the whitespaces are replaced by underscore _. For example: <p,c=warning_not_important creates <p class="warning not important">. If an underscore is part of the attribute-value, use escape \_. An equal-sign needs te be escaped as well: \=.

Tag-names and attribute-names should be written in smallcase characters.

Boolean attributes are inserted as for example ...,async=,... (equal sign, but no value).

Empty elements? <br > Simple.

To write < or > as regular content use the escape sign: \< , \> .

To summarize: the opening of an element (opening tag with attributes) is one 'word' without whitespace.

1.1 Anonymous tags and attributes

In some cases the opening tag needs no name:

The anonymous opening tag < is interpreted as:

  • a <li if the element is a child of <ul or <ol
  • a <td if the element is a child of <tr
  • a <p in all other contexts

There are also some anonymous attributes:

  • ,http://demo.org becomes href="http://demo.org when used in element a , as in <a,http://demo.org
  • ,demo-picture.png becomes src="demo-picuture.png" when used in element img , as in <img,demo-picture.png
  • ,red becomes class="red" in all other cases

Anonymous attributes can be combined with other (short) attributes as in <img,demo.png,a=nice_picture

Anonymous attributes can be used in anonymous tags.

1.2 More on the closing tag

It is allowed to use the name of the element in the closing tag, for example div> . It is not a bad habit to use this option when there is a lot of (complex) content between the opening and closing tag. Some favorites: table>, ol> and ul>, section>, ... In fact, it facilitates the setting of 'folding'-delimeters in editors like Notepad++.

Although a closing > or (for example) div> should in principle be a separate word (preceded and followed by whitespace), following habits are tolerated:

  1. a string with multiple closing tags, like: >>> A good hh-reader should split them up in seperate >
  2. a punctuation which sticks to and follows > or (for example) div> , like in >? or >.
  3. closing and opening tag sticking together, like in <b bold ><i italic ><b bold >. This produces: bolditalicbold.

There's a limit: do not combine option 1 with others. Do not apply 2 in de middle of a sticking 3.

1.3 HTML, SVG, script- and pre-elements

Users can insert HTML, SVG, script-elements and pre-elements, using delimeters which instruct the parser to copy as-is all the lines between them. The delimeters start with an exclamation mark, followed by at least five hyphens: !-----
These tokens should be inserted at the beginning of a new line.

Example:

    
    <pre <code,language-python
    !------------------
        class Element():
            def __init__(self, tag="", parent=None):
                self.parent = parent
    !------------------
    > >

That's it for the basics.

2 At your service

2.1 User-defined tags

It happens that an author has to use the same tag with certain attributes and values regularly.

If for example we often use <code,c=language-python... we can reduce this to <PY ... if we have defined the PY-element as a replacement for <code,c=language-python.

The definition is made with: <PY:=code,c=language-python > , of course before it's first use in the text.

This 'element' instructs the parser to replace 'PY' with 'code,c=language-python'.

The use of such a 'PY' is demonstrated in following pre-element (python-code is colored with highlight.js (url 3) ):


     # demo.hh 2 demo.html
    
    import sys

    sys.path.append('../../')
    import hh

    config = {'sourcefile':'demo.hh', 'outputfile':None, 'toc':2, 'h1':0, 'numbers': 1, 'includes': 3 , 
              'css':'default', 'katex':1, 'highlight': 1, 'dir':'ltr', 'lang':'en', 'class': 'hh', 'view': False}
    
    doc = hh.Document(config)
    print(doc.dublin_core['dc.title'])
    print('DONE')

Another example. When a table uses td-elements where an attribute needs to be set to certain values. For example, in a table with Greek text and English translations, we have <td,lang=el and <td,lang=en

These can be replaced with <G and <E respectively, using following definitions:

<G:=td,lang=el >

<E:=td,lang=en >

Following table uses these user-defined elements:

Greek English
Ἐπίκουρος Epicurus
Ἕρμαρχος Hermarchus or Hermarch

We use capital characters for the user-defined tags, in order not to intervene with the regular tags (always smallcase). Do not use names starting with 'SET' or 'GET'. These are used for footnote e.a.

Question. If we have <PY:=code,c=language-python >, the class of PY (code) is set. But can I add more attributes while using PY. Yes. This works: <PY,i=script-A. It will create <code class='language-python' id='script-A'>. Demo:

This code tag has id-value 'script-A'.

2.2 Comments

It is possible to include comments, that is text which should be ignored by the reader-script and leaves no traces in the html-source

As in html you place a comment between <!-- and -->. However, the opening 'tag' should appear at the beginning of a new line, while the closing 'tag' should be the last 'word' in a line.

This can be used to 'comment out' part of the text.

If one wishes to insert comment in the html-source, use !---------------

2.3 Includes

We can build one HTML-document from multiple source-files, using the <INCLUDE element. In your main source-file write <INCLUDE:chapter1.hh > and the content of chapter1.hh will be parsed and inserted on that spot. Obviously, the hh-reader-scripts should be launched with a source-file that serves as the main hh-source. One include (say a chapter) can call another include (say a section).

There is a protection against circular includes (A.txt including B.txt and B.txt including A.txt). The hh-reader produces a warning at a certain level (include-depth), which is by default 4. You can choose: continue or not. See below: usage.

2.4 Automatic Numbering of elements (h1), h2, ...

There are two options:

  1. h1-elements are included in numbering (config['h1'] == True)
  2. h1-elements are not included (config['h1'] == False)

When option 2 is set in configuration (see below): all h2, h3, h4, h5 and h6 elements get an id, for example: id='section_2_1' if it is the first h3 under the second h2.

When option 1 is selected, the same h3 might have id="section_1_2_1", if found under the first h1.

Another couple of options

  1. Numbers are visible for the reader of your webpage (config['numbers'] == True)
  2. Numbers are not visible for the reader of your webpage (config['numbers'] == False)

True or False: the id's are always there as a href-target.

2.5 Table of contents

Use the empty element <TOC > to mark the position of the table of contents in the document.

Obviously, the contentlist will follow the options concerning visual numbering and inclusion of h1.

If config['toc'] == 2 the table will contain headers upto level h2.

config['toc'] has value 1 upto 6 or ... 0 (zero) which results in no table at all (with or without <TOC >).

2.6 Footnotes: automatic numbering

A footnote example [1] and a second footnote [2] .

The content of a footnote is wrapped in an element with tagname <SETF .

Each footnote has a label, for example <SETF:labelx .

In the text we create a link to this footnote with <GETF:labelx >

The hh-reader inserts a number at GETF, and the content of a SETF with the same label and number appears at the bottom of the html-page. The labels are used to create the links and backlinks.

The SETF-elements can be inserted anywhere in the hh-source, after or before the GETF-element with the same label.

We can refer multiple times to the same footnote, repeatedly using the same label, like in (see footnote [2] ) The backlink however refers to the first use of the footnote. We can use an id only once. So far we have valid HTML5 at WC3's nu validator (url 5)

You should indicate if and where the footnotes should appear in the html-file. Simply insert the empty element <NOTES > somewhere near the end of your main sourcefile.

If there are multiple <NOTES > in a hh-file (perhaps inserted by one or more <INCLUDE's, then only the last <NOTES > in the complete source will be used as location for the footnotes.

2.7 Labeled References

For references one can create a seperate list where the items are labeled

A paragraph with a reference [Parrot 1999]

Another paragraph with the second second reference [Parrot 2009]

You should indicate if and where the footnotes should appear in the html-file. Simply insert the element <REFS > somewhere near the end of your main sourcefile.

You can use <INCLUDE to load a database with all your favorite references as <SETR's, even if you use only a few of them. Only the used references will pop at the bottom of the html-file.

If there are multiple <REFS > in a hh-file (perhaps inserted by one or more <INCLUDE's, then only the last <REFS > in the complete source will be used as location for the footnotes.

2.8 Metadata

Search engines look for metadata in de head-element. With HaHa, it is easy to provide metadata in an international and widely used standard format: the Dublin Core Metadata Element Set (DCMS)

WikiPedia's 'Dublin Core' page (url 6) is a good/sufficient guide.

  1. dc.title – "A name given to the resource".
  2. dc.creator – "An entity primarily responsible for making the resource" (author).
  3. dc.contributor – "An entity responsible for making contributions to the resource".
  4. dc.publisher – "An entity responsible for making the resource available".
  5. dc.date – "A point or period of time associated with an event in the lifecycle of the resource".
  6. dc.description – "An account of the resource" (abstract).
  7. dc.subject – "The topic of the resource (keywords)".
  8. dc.coverage – "The spatial or temporal topic of the resource, the spatial applicability of the resource, or the jurisdiction under which the resource is relevant".
  9. dc.format – "The file format, physical medium, or dimensions of the resource".
  10. dc.identifier – "An unambiguous reference to the resource within a given context" (URL, ISBN).
  11. dc.language – "A language of the resource" (ISO code).
  12. dc.relation – "A related resource".
  13. dc.rights – "Information about rights held in and over the resource".
  14. dc.source – "A related resource from which the described resource is derived".
  15. dc.type – "The nature or genre of the resource".

In HaHa, you do not have te create seperate elements with metadata. You can reuse the content.

This page starts with: <h1,dc.title HH guide (version 0.4 beta) >

The title is saved in a metadata-dictionary and used in meta-elements (and in this case the title-element as well).

Other example. The small, table on top of this page, has hh-markup:


 <,dc.creator (wdj) >
<,dc.date 2-22-2022 >
<,dc.rights <a,https://creativecommons.org/licenses/by/4.0/ Creative Commons Licence CC BY 4.0 > >

You may wonder: what about the inline markup of our content? It is removed. The stuff in the metadata-dict is plain text.

HTML validators have problems with metadata, although we follow HTML5 standards (url 7) (follow link to MetadataExtensions page). Our main motivation, when using Dublin Core: Open Archives Initiative (url 8) and DOAJ (Directory of Open Access Journals) (url 9)

2.9 Printing: URL's

HaHa provides facilities to print the url's of used links at the end of a webpage. This list does not show up on screen, but when printing of page is requested, the list shows up (unless the css is altered).

The links in the list are numbered and a reference number, like (url 5), appears after the link in the text.

While printing, backlinks are made invisible with css.

3 Python script hh.py

3.1 Usage

Download zip-file at github.com/notsue/HaHa (url 12) (press red code-button, and select 'download zip')

For practical learning and experimenting the file docs/hh_demo/hh_demo.hh can serve as a good start.

In a terminal window move to the directory docs/hh_demo/

Try the script: >python ../../hh.py --version

Then: >python ../../hh.py -h

If this produces normal answers, you can try to convert demo.hh into demo.html yourself:

>python ../../hh.py demo.hh -K -H

Javascripts: The option -K activates Katex, the option -H activates Highlight (you should be online)

The html-output demo.html is stored in the same directory

If you add option -v to the command: the script will ask if you want to view the html in your favorite browser.

The option -n activates the numbering of titles.

Check the metadata in the html-output. Observe the effect of the default css on printing.

Observe that your command-options are stored in a comment at the beginning of the html-file.

docs/hh_demo/demo.py demonstrates how to call hh.py from another script. It allows us to reduce command to:

>python demo.py

Send comments, tips, wishes and questions to notsue at protonmail dot com . Do not compare with markdown, ReST, etc. HaHa is not another lightweight markup. It's 'dirty' HTML. As dirty as possible.

3.2 What's next

With version 0.4 the markup became stable. This will not change in future versions. Perhaps more bells and whistles are added, but that will not harm the compatibility of older HaHa-source.

The handling of errors in the hh-source could be better. A clever user should however exploit the INCLUDE-mechanism which provides a means to test a hh-source piece by piece. When all parts are ok, you can cut and past them in one file.

3.3 Adaptations

You can change the script in order to include more or other css and javascript.

Whatever: do not touch the class Element.

In the class Document you should not touch the function read_include()

We have inserted the marker #OPTION: in places where changes might contribute to your local needs.
